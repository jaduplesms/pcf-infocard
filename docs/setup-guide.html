1 Overview

InfoCard is a compact, read-only PCF virtual control for model-driven Power Apps. It replaces the standard multi-field form layout with a single, slot-based card that consolidates header, contact, detail, grid, and tag information into one component.

It is designed for mobile-first scenarios where screen real estate is limited — Field Service technicians on phones, tablets in the field, or embedded quick-view forms. Related entity data is fetched automatically using the @fieldName convention, eliminating the need for multiple form lookups.

2 Installation

Option A — Direct push (development)

# From the InfoCardControl directory cd InfoCardControl pac pcf push --publisher-prefix cli

This pushes the control directly to your connected Dataverse environment. Use pac auth list to verify your target org.

Option B — Solution import (production)

  1. Build the solution: msbuild /t:build /restore /p:configuration=Release from the InfoCardSolution directory.
  2. Navigate to make.powerapps.com → Solutions → Import.
  3. Upload the .zip from InfoCardSolution/bin/Release/.
  4. Publish all customizations after import completes.
Version Bump Required
Always increment the version in ControlManifest.Input.xml before pushing. The Field Service mobile app caches PCF bundles aggressively — a stale version attribute means your update will not appear on devices until the cache expires.

3 Configuration

InfoCard is added to a model-driven form like any PCF control:

  1. Open the form editor for your target entity (e.g., Bookable Resource Booking).
  2. Add or select a text field on the form. This will become the titleField (bound property). It must be named value internally — PCF enforces this for model-driven forms.
  3. Open the field's Properties panel → Controls tab → Add Control → search for InfoCard.
  4. Configure the Layout Mode (smart/contact/compact) and bind each slot property to the appropriate column or @fieldName value.
  5. Enable the control for Phone and/or Tablet form factors.
  6. Save and Publish the form.
Bound Property Constraint
The single bound property (titleField) must map to a real column on the form entity. This is how the control anchors itself to the record. All other properties are input (static or expression).

6 Related Fields (@ syntax)

The @fieldName convention lets you pull data from a related entity without creating additional lookups on the form. When InfoCard sees a property value starting with @, it resolves the field through the titleField lookup.

1-hop: Direct lookup field

The titleField is bound to a lookup column (e.g., msdyn_workorder on the Booking entity). Any @fieldName fetches that column from the looked-up entity.

2-hop: Through a second lookup

If the resolved field on the Work Order is itself a lookup (e.g., msdyn_serviceaccount → Account), InfoCard can follow that second hop to fetch a field from the Account entity.

How it works
When the title lookup resolves a Work Order, and the @msdyn_serviceaccount value is itself a lookup, InfoCard issues a second WebAPI call to the Account entity to fetch additional fields like telephone1. This is automatic — no extra configuration needed beyond setting the @ property values.

7 Field Binding ($ syntax)

The $columnName syntax is used in the PCF dev harness for testing. When a property value starts with $, the harness reads the value from the mock data.json file using the column name after the prefix.

// In test-scenarios.json or harness config: titleField = $msdyn_name // Reads data.json["msdyn_name"] phoneField1 = $telephone1 // Reads data.json["telephone1"] gridField1 = $msdyn_starttime // Reads data.json["msdyn_starttime"]

This lets you test the control with realistic entity data without deploying to Dataverse. The $ prefix is harness-only — in production, properties are bound to actual form columns through the form editor.

8 Tips & Best Practices

hideEmptyFields

Enabled by default. When a slot has no value (null, empty string, or placeholder), the row is hidden entirely. Disable this only if you want to show placeholder text for all configured slots.

Recommendation
Keep hideEmptyFields = true for mobile scenarios. It prevents the card from showing “—” placeholders that waste vertical space on small screens.

startExpanded

Only applies to the smart layout. When true (default), the card shows all sections on load. Set to false to start collapsed — users see only header, contact chips, and tags, and can tap the chevron to expand.

Recommendation
Use startExpanded = false when the card is placed alongside other controls on a busy form. Technicians can expand when they need the detail, keeping the initial view compact.

General Tips

  • Always bump the version in ControlManifest.Input.xml before each pac pcf push.
  • Use @fields sparingly — each related lookup costs a WebAPI call. Keep to 1-2 hops max.
  • Title lookups are clickable by default — tapping opens the related record in a new form.
  • Phone/email chips invoke native device actions (tel:, mailto:) on mobile.
  • Address + lat/lng opens Google Maps. Ensure both coordinates are bound for the map link to appear.
  • CSS is fully inline — no external stylesheets, no class name collisions with Dynamics 365 forms.
  • Offline support: The card renders whatever data is available. Related @field fetches fail gracefully when offline.
