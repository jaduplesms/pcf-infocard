## What This Is

A Power Apps PCF (PowerApps Component Framework) control called **InfoCard** that renders multiple read-only form fields in a compact, mobile-optimized card layout for Dynamics 365 Model-Driven Apps. Built for Field Service Mobile technicians who spend excessive time scrolling through forms with many read-only fields.

## Build & Deploy Commands

```bash
# Build the PCF control (from InfoCardControl/)
cd InfoCardControl
npm run build

# Start the PCF test harness (browser-based preview)
npm run start

# Regenerate TypeScript types from manifest after changing ControlManifest.Input.xml
npm run refreshTypes

# Build the Dataverse solution package (from InfoCardSolution/)
cd InfoCardSolution
dotnet build --configuration Release
# Output: InfoCardSolution/bin/Release/InfoCardSolution.zip

# Deploy to Dynamics 365 environment (pac CLI must be authenticated)
pac solution import --path InfoCardSolution/bin/Release/InfoCardSolution.zip --environment "https://jaduplesft.crm11.dynamics.com/" --publish-changes
```

Auth profile for pac CLI: `jaduples@DynamicsFastTrack.onmicrosoft.com` targeting `jaduplesft.crm11.dynamics.com` (profile [1]).

## Architecture

**v2 (current, deployed)** uses a slot-based approach. The v1 approach (JSON payload via companion form script in `DESIGN.md`, `InfoCard_FormScript.js`) is superseded.

### Two projects, one solution

- **`InfoCardControl/`** — PCF control project (React virtual control, namespace `Contoso`, constructor `InfoCard`)
- **`InfoCardSolution/`** — Dataverse solution wrapper (publisher prefix: `cli`), packages the control into a `.zip` for import

### PCF control source (InfoCardControl/InfoCard/)

- **`ControlManifest.Input.xml`** — Defines 28 typed property slots in groups + 3 config properties. This is the source of truth for what the control accepts.
- **`index.ts`** — PCF lifecycle class. Reads bound slot values from `context.parameters`, resolves display values (handles lookups, formatted values, nulls), resolves Fluent Design Language theme tokens, and passes structured `InfoCardData` to the React component.
- **`InfoCard.tsx`** — React component with three layout renderers (`SmartCardLayout`, `ContactCardLayout`, `CompactFormLayout`) sharing common sub-components (ActionButton, TagChip, FieldRow). Includes inline SVG icons — no icon library dependency.
- **`generated/ManifestTypes.d.ts`** — Auto-generated by `npm run refreshTypes`. Regenerated on build. Don't hand-edit (build will overwrite).

### Slot system

Developers bind entity fields to typed slots in the form designer. The slot name determines the rendering role:

| Group | Slots | Rendering |
|-------|-------|-----------|
| Header | `titleField`, `subtitleField1-3` | Card title (no label) + dot-separated subtitle line |
| Actions | `phoneField1-2`, `emailField`, `webField` | Tappable icon buttons (`tel:`, `mailto:`, `https:` links) |
| Details | `detailField1-5` | Icon + text rows (Smart/Contact) or label:value rows (Compact) |
| Grid | `gridField1-10` | 2-col key:value (Smart), tags (Contact), labeled rows (Compact) |
| Tags | `tagField1-5` | Compact chip/pill strip |

Config properties: `layout` (smart/contact/compact), `hideEmptyFields`, `showCardBorder`.

### Layout modes

All three layouts render the same five zones (header, actions, details, grid, tags) but with different visual treatment. Zones auto-hide when empty. See `DESIGN_v2.md` section 5 for the full zone-mapping table.

- **Smart Card** — Contact bar with phone number text, 2-col grid, detail rows with auto-detected icons
- **Contact Card** (default) — Icon buttons top-right, icon+text detail rows, grid falls through to tags. Most compact.
- **Compact Form** — Section labels (Contact/Details/Info), label:value rows. Preserves "form feel."

### Theme resolution

`index.ts` reads `context.fluentDesignLanguage.tokenTheme` (Preview API) and maps tokens to `InfoCardTheme`. Falls back to hardcoded light theme in `defaultTheme`.
