InfoCard v2 — Slot-Based Configuration

The developer binds entity fields to typed slots in the form designer. The slot name determines how the field renders. A layout property controls overall arrangement.

Property Slots

Slot Count Renders as Example binding
titleField 1 Card header text, no label name (Account Name)
subtitleField1–3 3 Subtitle line, dot-separated primarycontactid, msdyn_serviceterritory
phoneField1–2 2 Phone icon button (tap to call). Value visible in A, tooltip in B/C telephone1, telephone2
emailField 1 Email icon button (tap to send) emailaddress1
webField 1 Globe icon button (tap to open) websiteurl
detailField1–5 5 Icon + text row (A/B) or label:value row (C) address1_composite, description
gridField1–10 10 2-col key:value grid (A) / tags (B) / label:value rows (C) transactioncurrencyid, msdyn_taxcode
tagField1–5 5 Compact chip/pill — all layouts statuscode, industrycode

Property Type Options Default
layout Enum Smart Card / Contact Card / Compact Form Contact Card

hideEmptyFields TwoOptions Yes / No Yes

How Slots Map to Zones per Layout

Each slot group always renders in the same card zone. The layout controls the visual treatment of that zone.

Slot Group Zone A: Smart Card B: Contact Card C: Compact Form
titleField Header Large name Large name Name in header bar
subtitleField* Header Dot-separated line Link-styled under name Not shown (use grid instead)
phoneField* Actions Contact bar: icon + number Icon button top-right Icon button in header bar
emailField Actions Contact bar: mail icon Icon button top-right Icon button in header bar
webField Actions Contact bar: globe icon Icon button top-right Icon button in header bar
detailField* Details Icon + text rows Icon + text rows Label:value rows
gridField* Grid 2-column key:value cells Falls through to tags Label:value rows with section header
tagField* Tags Chip strip (if used) Chip strip at bottom Chip strip at bottom

Key Design Decisions

  • Empty fields always hidden — no "---" placeholders (configurable via hideEmptyFields for edge cases)
  • Same field can be in multiple slots — e.g. phone in both phoneField1 (action icon) and detailField2 (visible number)
  • Field labels auto-derived from the bound field's display name — no manual labelling needed
  • Action slots use typed of-type — SingleLine.Phone, .Email, .URL — so the form designer shows only relevant fields in the picker
  • 28 bound slots total — more than enough for any entity, unused slots are just empty
  • No companion form script needed — all data comes through bound properties directly
  • gridField in Layout B — renders as tags (no separate grid zone), keeping it ultra-compact
